# Research Session Conventions

This document is the canonical contract for research-mode sessions in this repo. Every research skill (`setup-task`, `plan-experiment`, `setup-methods`, `setup-analyses`, `run-experiment`, `interpret-experiment`, `replicate-paper`, `document-issues`) reads this file at Step 0 and follows the rules here.

If a downstream skill's behavior conflicts with this document, this document wins. Update it (and propagate the change to consuming skills) rather than diverging.

## Session naming

```
agent_logs/{YYYY-MM-DD}--{topic}--{adj-noun}/
```

Three components, separated by `--`:

| Component | Source | Example |
|---|---|---|
| `YYYY-MM-DD` | UTC date (no time-of-day) | `2026-05-02` |
| `topic` | 1‚Äì3 word slug describing the research focus | `locate-weekdays` |
| `adj-noun` | LLM-friendly disambiguator from the SKILL.md wordlist | `pensive-mongoose` |

Full example: `agent_logs/2026-05-02--locate-weekdays--pensive-mongoose/`

The topic is **required**. Placeholders like `auto`, `untitled`, `test`, `tmp` are rejected.

## Core invariant: agents write only to the session dir

Every research skill must confine its writes to `agent_logs/{SESSION_NAME}/`. Skills do not edit files under `causalab/`, `demos/`, `scripts/`, or any other shipped path during a research workflow.

The single permitted exception is **manual promotion** ‚Äî moving a stabilized session-local prototype into the shipped `causalab/` tree. The promotion paths are:

| Session-local source | Shipped target |
|---|---|
| `code/methods/<name>/` | `causalab/methods/<name>/` |
| `code/analyses/<name>/` | `causalab/analyses/<name>/` |
| `code/configs/analysis/<name>.yaml` | `causalab/configs/analysis/<name>.yaml` |
| `code/configs/runners/<group>/<name>.yaml` | `causalab/configs/runners/<group>/<name>.yaml` |
| `plan/set_up_task_draft.md` | `causalab/tasks/<name>/set_up_task.md` (+ task package) |

Promotion is a `git mv` step (or, for tasks, the explicit Step 2 approval checkpoint in `/setup-task`'s outer flow that scaffolds the full task package). Skills never auto-promote without an approval gate.

Any code or artifacts a skill generates that get executed in the user's environment (notebooks, scripts) must follow the same rule: writes target session-local paths only, never the shipped `causalab/` tree.

## Directory layout

```
agent_logs/{SESSION_NAME}/
‚îú‚îÄ‚îÄ README.md         single-paragraph intent + layout cheatsheet
‚îú‚îÄ‚îÄ issues.md         top-level issue log (managed by /document-issues)
‚îú‚îÄ‚îÄ plan/             pre-execution artifacts
‚îú‚îÄ‚îÄ run/              execution artifacts (resolved cfg, run.log, slurm logs)
‚îú‚îÄ‚îÄ result/           post-execution narrative
‚îú‚îÄ‚îÄ code/             session-local Python + Hydra (via /setup-methods, /setup-analyses, /run-experiment)
‚îî‚îÄ‚îÄ artifacts/        raw experiment outputs (mirrors {task}/{model}/... layout)
```

### What goes in `plan/`

Pre-execution thinking. Written by `/setup-task`, `/run-experiment` (early steps), `/replicate-paper` (early steps).

- `research_objective.md` ‚Äî the question being investigated, success criteria, scope.
- task-specification drafts under discussion (before they're promoted to `causalab/tasks/<task>/set_up_task.md` via `/setup-task`'s approval checkpoint)
- approval-checkpoint logs from interactive sessions

Runner configs are written directly to `code/configs/runners/<group>/<name>.yaml` ‚Äî they don't go through a `plan/` draft step (see "What goes in `code/`" below).

### What goes in `run/`

Execution artifacts. Written by `/run-experiment` (Step 4+), `/replicate-paper` (run phase).

- `<runner>_resolved.yaml` ‚Äî output of `scripts/run_exp.sh <runner> --cfg job`, captured before the run. Includes the session-scoped `experiment_root` override.
- `run.log` ‚Äî stdout/stderr from the run.
- slurm job IDs, sbatch logs, retry notes.

Raw artifacts (tensors, scores, plots) land under `{session}/artifacts/{task}/{model}/...` ‚Äî see *Output routing* below. There is no `artifact_pointers.md` file; the artifacts are right there to browse.

### What goes in `result/`

Post-execution narrative. Written by `/interpret-experiment`, `/replicate-paper` (interpretation phase).

- `REPORT.md` ‚Äî single consolidated interpretation written by `/interpret-experiment`. Cross-run and cross-session comparison is a section *inside* this file, not a separate document.
- `figures/` ‚Äî curated plots / tables copied or symlinked from `artifacts/` and embedded in `REPORT.md` via relative markdown links.
- hand-off notes for downstream sessions (e.g. "the locate run picked layer 14; subspace runs should target it") ‚Äî either a section in `REPORT.md` or a sibling file.

### What goes in `code/`

Session-local Python and Hydra config generated by `/setup-methods`, `/setup-analyses`, and `/run-experiment`. Layout:

```
code/
‚îú‚îÄ‚îÄ configs/
‚îÇ   ‚îú‚îÄ‚îÄ analysis/<name>.yaml          # `# @package <name>` Hydra config (mounts at cfg.<name>)
‚îÇ   ‚îî‚îÄ‚îÄ runners/<group>/<name>.yaml   # runner configs (chains analyses; references analysis/<name>)
‚îú‚îÄ‚îÄ analyses/
‚îÇ   ‚îî‚îÄ‚îÄ <name>/                       # Hydra entry point (main.py + README.md)
‚îî‚îÄ‚îÄ methods/
    ‚îî‚îÄ‚îÄ <name>/                       # reusable primitive (no Hydra, no I/O paths)
```

The runner picks these up automatically when `CAUSALAB_SESSION_CODE=1` (set by `scripts/run_exp.sh` whenever `--experiment-root` lives under `agent_logs/`). Concretely the wrapper prepends `${SESSION_DIR}/code/` to `PYTHONPATH` so `import analyses.<name>` and `import methods.<name>` resolve, adds `${SESSION_DIR}/code/configs/` to Hydra's search path so `- analysis/<name>` defaults entries find the session-local YAML, and discovers runner configs under `${SESSION_DIR}/code/configs/runners/` (see `scripts/run_exp.sh:99-141`).

Session-local code follows the same layering rules as shipped code (see `ARCHITECTURE.md` ¬ß3 invariants 1‚Äì5, 7, 11, 12). Promote to `causalab/methods/` or `causalab/analyses/` manually (`git mv` + remove from `code/`) once the prototype stabilizes ‚Äî there is no automated promotion step.

If a session-local `<name>` collides with a shipped `causalab/{methods,analyses}/<name>/`, the shipped one wins (the dispatcher tries `causalab.analyses.<name>` first). Both `/setup-methods` and `/setup-analyses` refuse colliding names at scaffolding time.

### What goes at the top level

- `README.md` ‚Äî written once at session creation; minor edits as scope evolves.
- `issues.md` ‚Äî the only issue log for the session. Spans all phases ‚Äî forcing it into a subfolder creates a nuisance every time you remember an issue from earlier in the workflow. Managed by `/document-issues`.
- `paper_data/` ‚Äî only present when the session is a paper replication. Holds the paper PDF, downloaded datasets, and other source material from the paper's repo or supplementary materials. Top-level (parallel to `plan/`/`run/`/`result/`) because it is *input* source data, not session output. Each replication session re-fetches its own copy.

## Output routing

Research-mode runs override `experiment_root` so all generated artifacts land inside the session, not in the global `artifacts/` tree.

| Mode | `experiment_root` resolution |
|---|---|
| Dev (no active session, no override) | `artifacts/{task}/{model}` (Hydra default in `causalab/configs/base.yaml`) |
| Research (session active) | `agent_logs/{session}/artifacts/{task}/{model}` |

The single knob is `cfg.experiment_root` (declared in `base.yaml`, consumed by every runner, analysis, and method). Research skills pass it as a Hydra override when launching the runner:

```bash
SESSION=$(cat agent_logs/.current)
./scripts/run_exp.sh --experiment-root "agent_logs/$SESSION/artifacts/<task>/<model>" <runner>
```

The override is captured in `run/<runner>_resolved.yaml` so the run is fully reproducible from the session bundle alone.

If the runner config sets `task.variant`, `apply_experiment_root_variant` (in `causalab/io/configs.py`) appends `/{variant}` to whatever root you supplied ‚Äî both modes get a consistent `‚Ä¶/{task}/{model}/{variant}` layout.

### Cross-session reuse

Self-contained sessions are the gold standard. Each session re-runs its own upstream stages (baselines, activations, counterfactuals) under its own `artifacts/`. There is **no automatic fallback** to the global tree or to another session.

If the user explicitly instructs reuse ("reuse activations from session X" or "use the global baseline"), point `experiment_root` at the source path or copy/symlink the needed subtree into the new session before launching. Either way, record the reuse in `plan/research_objective.md` so the session's provenance stays legible.

## Active-session detection protocol

The marker file `agent_logs/.current` (plain text, no extension) contains the directory name (not the full path) of the active session.

```
$ cat agent_logs/.current
2026-05-02--locate-weekdays--pensive-mongoose
```

### Step 0 inline check (every research skill must run this)

```
if $CAUSALAB_AUTONOMOUS == "1":
    # Autonomous mode (eval harness): the harness is responsible for
    # setting .current before invoking the skill. If unset, the skill
    # may bootstrap one inline rather than failing ‚Äî but this should
    # be rare; the harness path normally pre-creates the session.
    pass
else:
    # Interactive mode
    if not exists("agent_logs/.current"):
        tell user: "No active research session. Run /research-session first."
        stop
    session_name = read("agent_logs/.current").strip()
    if not isdir(f"agent_logs/{session_name}"):
        tell user: "Active session marker points at a missing directory.
                    Run /research-session to create a fresh session."
        stop
    session_dir = f"agent_logs/{session_name}"
    proceed using session_dir
```

The `.current` file is robust across machines (no symlinks), explicit (not "whichever dir has the newest mtime"), and editable ‚Äî a user can manually swap to a different session by overwriting it.

### Multi-agent isolation

`.current` is a **single global marker per checkout**. If two Claude agents share one checkout, they share one `.current` and can clobber each other's session pointer mid-workflow. The supported escape hatch is `/research-session --worktree`, which calls `EnterWorktree` before bootstrapping the session dir. Each worktree is a separate checkout with its own `agent_logs/.current`, so per-agent isolation is automatic and no skill-level changes are required.

When to use it:

- Running multiple Claude instances against the same repo at the same time ‚Üí use `--worktree` for every concurrent session.
- Running a single agent ‚Üí leave it off; the main checkout's `agent_logs/` stays browsable from the IDE and the HTML-preview server without redirection.

Tradeoffs (covered in `SKILL.md`'s argument docs): cross-session comparison requires merging the worktree branch back or addressing files via `<repo>/.claude/worktrees/<session-name>/`; tooling rooted at the main checkout (e.g. the Cursor HTML-preview server on :8000) does not see worktree files until merged or symlinked.

Manual variant: `git worktree add <path>` followed by `EnterWorktree` with `path:<that path>` enters an existing worktree without creating a new one ‚Äî useful when the worktree was set up by something other than this skill.

## Required-reading snippet for downstream skills

Every research skill SHOULD have this near the top of its SKILL.md:

```markdown
## Required Reading

Before running this skill, read:
- `.claude/skills/research-session/CONVENTIONS.md` ‚Äî research-session layout and active-session protocol.

If no active session exists (`agent_logs/.current` missing or stale), instruct the user to run `/research-session` first and stop.
```

## What does NOT go in the session dir

- **Stable code that ships with the repo** ‚Äî task modules go in `causalab/tasks/<task>/`, configs go in `causalab/configs/`, library code in `causalab/{neural,methods,analyses,io,runner}/`. Session-local *prototypes* generated by `/setup-methods` or `/setup-analyses` live under `code/` and should be promoted manually once stable.
- **Notebooks for permanent demos** ‚Äî those go in `demos/analyses/<name>/demo.ipynb` per ARCHITECTURE.md ¬ß3 invariant 10. The session dir may hold ad-hoc exploratory notebooks under `result/` if they're part of the session writeup.
- **Dev-mode runs** ‚Äî runs launched outside an active session keep using the global `artifacts/` default per ARCHITECTURE.md ¬ß3 invariant 7. Don't redirect them into a session after the fact.

## Per-skill mapping

| Skill | Reads from session | Writes to session |
|---|---|---|
| `/setup-task` | session dir for context | `plan/` (spec drafts, approval logs) |
| `/plan-experiment` | session dir for context | `plan/` (`RESEARCH_OBJECTIVE.md`, `PLAN.md`, approval logs) ‚Äî **gates on existing plan** |
| `/setup-methods` | `plan/PLAN.md` (one or more custom-method specs) | one `code/methods/<name>/` per spec |
| `/setup-analyses` | `plan/PLAN.md` + `code/methods/` (one or more custom-analysis specs) | one `code/analyses/<name>/` + `code/configs/analysis/<name>.yaml` per spec |
| `/run-experiment` | session dir for context, `code/` for session-local code | `code/configs/runners/<group>/<name>.yaml` (runner config), `run/` (resolved cfg, logs), `artifacts/` (raw outputs via the `experiment_root` override) |
| `/interpret-experiment` | `plan/`, `run/<runner>_resolved.yaml`, `code/` (session-local methods/analyses), `artifacts/{task}/{model}/...` | `result/REPORT.md`, `result/figures/` |
| `/replicate-paper` | session dir for context | `paper_data/` (downloaded sources), `plan/objective_input_draft.md`, `plan/paper_context.md` (sidecar ‚Äî gates only its own re-invocation, not `/plan-experiment`), `plan/approval.log`. Downstream artifacts (`RESEARCH_OBJECTIVE.md`, `PLAN.md`, `run/`, `artifacts/`, `result/`) are produced by the skills `/replicate-paper` hands off to (`/plan-experiment`, `/run-experiment`, `/interpret-experiment`). |
| `/document-issues` | ‚Äî | `issues.md` (top level) |

## One session, one plan

A session scopes a single investigation. It must hold at most one plan-level artifact set:

- `{plan/RESEARCH_OBJECTIVE.md, plan/PLAN.md}` ‚Äî written by `/plan-experiment` (also produced when `/replicate-paper` hands off to `/plan-experiment`).
- `plan/paper_context.md` (replication sidecar) ‚Äî written by `/replicate-paper`. **Not** treated as plan-level for gate purposes: by design `/replicate-paper` writes this file and immediately hands off to `/plan-experiment`, so its presence is the expected entry state for that chain. `/replicate-paper` itself still refuses if it sees a sibling `paper_context.md` from a prior run; `/plan-experiment` ignores the sidecar and only gates on `RESEARCH_OBJECTIVE.md` / `PLAN.md`.

Never two of either. If a user invokes a plan-producing skill in a session that already has a plan, the skill must stop and offer two choices:

1. **Switch sessions** ‚Äî run `/research-session` to start a fresh session, then re-invoke the plan skill there.
2. **Hold off** ‚Äî keep the existing plan and exit without changes.

There is no in-place "overwrite" option: if the user wants a new plan, the right answer is a new session. This keeps each session's provenance legible and prevents two competing objectives from sharing one `artifacts/` tree.

In autonomous mode (`CAUSALAB_AUTONOMOUS=1`), the harness is responsible for creating a fresh session per plan. If the gate triggers in autonomous mode, the skill exits non-zero with a clear error rather than guessing the user's intent.

Drafts that are part of *executing* an existing plan ‚Äî `plan/set_up_task_draft.md` (from `/setup-task`), `plan/<name>_draft.yaml` (from `/run-experiment`) ‚Äî are not plan-level artifacts and do not trigger the gate.

## Re-invocation

`/research-session` invoked while a session is already active prompts the user with three choices:

1. **New** ‚Äî replace `.current`, create a fresh session.
2. **Continue** ‚Äî keep using the existing session (no-op).
3. **Abort** ‚Äî exit without changes.

In autonomous mode (`CAUSALAB_AUTONOMOUS=1`), always start a new session.

## Cleanup and archival

Sessions are not auto-deleted. To archive completed sessions:

```bash
mkdir -p agent_logs/archive
mv agent_logs/2026-05-02--<topic>--<adj-noun> agent_logs/archive/
```

If you archive the active session, also clear `.current`:

```bash
rm agent_logs/.current
```

The next research skill invocation will then prompt for `/research-session`.
